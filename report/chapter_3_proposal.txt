CHAPTER 3: IMPLEMENTATION, EXPERIMENTATION AND EVALUATION

3.1. Setting Up and Configuring Snort 3 Detection System
Snort 3 is an advanced, open-source network intrusion prevention system (NIPS) and network intrusion detection system (NIDS) capable of performing real-time traffic analysis and packet logging on IP networks. In this project, Snort 3 serves as the primary signature-based detection engine, running in parallel with our Machine Learning models to provide a comprehensive hybrid defense mechanism. Its multi-threaded architecture allows for high-throughput packet processing, making it ideal for monitoring our network interfaces.

3.1.1. Deploying Snort 3 on Docker Environment
The deployment process utilizes containerization technology to ensure environment consistency, isolate dependencies, and facilitate ease of scalability across multiple target machines. We use Docker Compose to orchestrate the Snort 3 container. 
Key implementation details include:
- Image Selection: Utilizing a specialized Snort 3 image (e.g., cisco/snort3) optimized for network traffic analysis.
- Volume Mapping: Mapping local configuration directories (`snort3/config/`) containing files like `snort.lua` and `local.rules`, alongside log directories (`/var/log/snort`), to the container. This ensures persistence of alerts and allows for dynamic rule updates without rebuilding the container.
- Network Configuration: The container is configured to use the host's network stack (host network mode) or a bridge network with the interface set to promiscuous mode. This allows Snort to monitor all traffic traversing the host's physical network interface, rather than just traffic destined for the container.
- Output Configuration: We modified the `snort.lua` configuration file to explicitly enable the `alert_json` plugin. This is a critical step, ensuring that all triggered alerts are exported in a structured JSON format (`alert.json`), which is easily parsed and ingested by the Python bridge (`snort_bridge.py`) and downstream systems like Logstash.

3.1.2. Configuring Rules to Detect Various Attack Types
The effectiveness of the signature-based detection component relies entirely on a robust and accurate set of rules. We implemented 23 custom rules in `local.rules` specifically targeting common threats:
- DDoS & DoS Detection: We configured rules to track the rate of SYN packets originating from a single source IP. By setting a `detection_filter`, the system triggers an alert if the threshold of SYNs per second is exceeded, indicating a potential SYN flood attack.
- Port Scanning: We implemented rules to detect high-frequency connection attempts to multiple disparate ports across the network within a short time frame, indicative of reconnaissance tools like Nmap.
- Web Attacks: Signatures were crafted to identify malicious payloads within HTTP traffic. This includes detecting common SQL injection strings (e.g., `UNION SELECT`, `OR 1=1`) in URIs or POST bodies, as well as detecting Cross-Site Scripting (XSS) script tags.
- Bruteforce: Rules were established to monitor repeated failed login attempts for specific services, such as FTP (port 21) or SSH (port 22), by tracking connection rates and specific response codes.

3.2. Integrating ELK Stack for Log Processing and Visualization

3.2.1. Configuring Logstash to Collect Data from Snort
Logstash acts as the robust data processing pipeline bridging Snort's raw JSON logs and the Elasticsearch storage engine.
- Input Stage: Logstash is configured with a file input plugin to continuously monitor and tail the `alert.json` file generated by Snort 3.
- Filter Stage: The `json` filter is applied to parse the raw log lines into distinct, queryable fields (e.g., `source_ip`, `dest_ip`, `alert_msg`, `priority`, `timestamp`). Furthermore, we utilize the `geoip` filter plugin to automatically cross-reference source IP addresses with a geolocation database, appending geographical coordinates to the log entry.
- Output Stage: The enriched JSON documents are forwarded to the Elasticsearch cluster, indexed under a specific pattern (e.g., `snort-alerts-*`), optimized for time-series data storage and rapid retrieval.

3.2.2. Building Monitoring Dashboard on Kibana
A specialized dashboard was created in Kibana to provide administrators with immediate, visual oversight of network security events:
- Real-time Alert Counters: Big-number visualizations displaying the total number of threats detected within the last 15 minutes, 1 hour, and 24 hours.
- Attack Category Distribution: Pie charts and donut charts illustrating the percentage breakdown of attack types (DoS vs. Web Attacks vs. Port Scans), allowing for quick threat landscape assessment.
- Geographic Map: A coordinate map leveraging the GeoIP data to show the physical origins of attacking IPs, highlighting potential regional threat concentrations.
- Detailed Log Table: A highly granular, searchable data table allowing administrators to drill down into specific packet details, filter by source IP or priority level, and conduct in-depth forensic analysis.

3.3. Building Real-time Monitoring System (Flask Dashboard)

3.3.1. Implementing REST API for Agent Requests
The Flask server (`app.py`) serves as the central command hub for our hybrid IDS architecture. 
- `/api/monitor`: A high-performance, asynchronous endpoint designed to receive pre-processed network feature vectors from `live_agent.py` scripts running on target machines. Upon receiving data, the server scales the features using a pre-trained `StandardScaler` and feeds them into the primary Random Forest model to predict anomalies in real-time.
- `/api/snort/alert`: A dedicated endpoint that receives forwarded JSON alerts from distributed Snort sensors (via `snort_sensor.py`). It updates internal in-memory data structures tracking sensor health and alert history.
- WebSocket Integration: We heavily utilize `Flask-SocketIO` to push newly detected anomalies (from the ML model) and signature matches (from Snort) immediately to all connected web clients via `socketio.emit()`. This completely eliminates the need for manual browser refreshing, achieving true real-time monitoring.

3.3.2. Designing Alert Dashboard (Web Interface)
The web interface (`index.html`) is built to provide a comprehensive "Command and Control" view:
- Live Alert Feed: A dynamic, scrolling list of the latest threats, color-coded by severity (Red for critical attacks, Yellow for suspicious/benign traffic).
- Model Performance View: A section displaying the pre-calculated accuracy metrics and comparison charts for the integrated ML models, loaded dynamically from `results.json`.
- Sensor Grid: A management view showing the real-time status of all distributed agents and sensors across the network. It tracks heartbeats to determine if a sensor is "Alive" or "Dead".
- Live Prediction Tool: An interactive section where users can manually input or modify network statistical features and test how the different loaded models (RF, SVM, KNN) classify the artificial traffic.

3.4. Machine Learning Training Results and Evaluation

3.4.1. Experimental Results on Test Dataset
The models were trained using the highly regarded CICIDS2017 dataset, which contains realistic network traffic spanning various attack categories (BENIGN, DoS, DDoS, PortScan, Brute Force, Web Attack, Bot). 
- Rigorous Data Preprocessing: The raw data underwent extensive cleaning in `preprocess.py`. Infinite and NaN values were replaced using median imputation. We engineered critical domain-specific features, including specific port booleans (HTTP, HTTPS, SSH, FTP, DNS) and interaction features like `flow_pkt_ratio` and `flow_byte_ratio` to capture behavioral anomalies.
- Advanced Feature Selection: To optimize training speed and reduce dimensionality, we applied Variance Thresholding, followed by a smart Correlation filter. We then used a combination of Random Forest Feature Importance and Mutual Information (normalized and weighted equally) to select the Top 40 most impactful features. We also implemented strict label leakage detection to ensure no target-correlated features biased the models.
- Handling Imbalance & Scaling: Because attacks are rare compared to normal traffic, we utilized SMOTE (Synthetic Minority Over-sampling Technique) with `sampling_strategy='not majority'` to balance the training set. Data was scaled using `MinMaxScaler` and normalized using `PowerTransformer` (Yeo-Johnson) prior to SMOTE to maintain accurate distance calculations.
- Split Strategy: The data was split into a 70% Training, 15% Validation, and 15% Testing ratio using stratified sampling to maintain class proportions across all sets.

3.4.2. Performance Comparison (Accuracy, F1-Score, Recall)
We evaluated three primary algorithms, tuning them to handle the massive dataset:
- Random Forest (RF): This model served as our primary engine. We utilized `RandomizedSearchCV` (cv=3, n_iter=10) to find optimal hyperparameters (e.g., n_estimators=300, max_depth=None/20/30). Due to the massive size of the SMOTE-augmented data (potentially millions of rows), we subsampled to 500,000 rows during cross-validation to prevent memory exhaustion (OOM). RF achieved the highest overall performance, with Accuracy frequently exceeding 99%. More importantly, it achieved the highest F1-Macro score, proving its robustness in correctly identifying both majority (Benign) and minority (Bot/Web Attack) classes.
- K-Nearest Neighbors (KNN): We configured KNN with `n_neighbors=15`, `weights='distance'`, and the `ball_tree` algorithm to improve stability and reduce noise from distant neighbors. While KNN showed excellent accuracy, it exhibited significantly higher latency during the prediction phase compared to tree-based models, making it less ideal for high-throughput real-time processing. We capped the training data at 200,000 rows for KNN to maintain reasonable training times.
- Support Vector Machine (SVM): We implemented `LinearSVC` wrapped within a `CalibratedClassifierCV` to extract probability scores (`predict_proba`). With `C=0.5` and `max_iter=5000`, the SVM performed well on binary separation but was slightly less effective in the complex multi-class boundaries of the CICIDS2017 dataset compared to the Random Forest ensemble.

For IDS evaluation, we heavily prioritized the **F1-Macro** score over simple accuracy to ensure minority attack classes were not overshadowed. We also monitored the Overfit Gap (Validation F1-Macro vs. Test F1-Macro) to ensure the models generalized effectively.

3.5. Attack Simulation Scenarios and Real-world Results

3.5.1. Simulating DDoS and Port Scan Attacks
To validate the system in a live environment, we deployed `live_agent.py` on target machines. This agent continuously captures interface statistics (bytes/s, packets/s) and TCP connection flags every 2 seconds, constructing a feature vector mirroring the CICIDS2017 format. We then conducted controlled attacks:
- Port Scan Scenario: Using the `nmap` tool, we initiated aggressive TCP SYN scans against the target machine. Within milliseconds, the Snort 3 container matched the signature of rapid sequential connection attempts, logging a 'Priority 3' alert to `alert.json`, which was instantly forwarded to the Flask dashboard. Simultaneously, the `live_agent` captured the spike in packet rates and flagged the traffic anomaly using its local ML inference.
- DDoS Scenario: We utilized `hping3` to flood the target with UDP and SYN packets. As the interface traffic metrics exponentially increased, the `live_agent` constructed a feature vector characterized by extreme `flow_pkt_ratio` and abnormal TCP flag ratios. The Random Forest model instantly classified the vector as 'DDoS' with high confidence (>95%), triggering a critical red alert on the WebSocket-powered web interface.

3.5.2. Analyzing Hybrid System Detection Capability
The hybrid approach demonstrated profound operational advantages during testing:
- High-Speed Known Threat Mitigation: Snort 3 acted as the first line of defense. By relying on exact signature matches (e.g., detecting a specific SQLi payload in an HTTP GET request), it operated with near-zero latency and provided explicit context regarding the attack type.
- Zero-Day and Behavioral Anomaly Detection: The Random Forest model acted as the second layer. When an attacker utilized a novel DDoS tool or mutated an existing exploit to bypass Snort's signatures, the ML model successfully caught the attack by recognizing the deviation from the established baseline of "normal" statistical network flow features.
This synergy significantly reduced the overall False Negative rate. The deterministic speed of Snort combined with the adaptable intelligence of Machine Learning created a deeply layered defense mechanism.

3.6. Evaluation of System Strengths and Limitations

Strengths:
- Comprehensive Detection Capability: The combination of signature-based (Snort) and anomaly-based (Machine Learning) detection drastically improves the system's ability to catch both known exploits and zero-day behavioral anomalies.
- Exceptional ML Performance: Extensive data preprocessing, rigorous feature selection, and the use of SMOTE resulted in highly robust models, with Random Forest achieving near-perfect F1-Macro scores, indicating excellent sensitivity to rare attack types.
- Real-Time Responsiveness: The architecture, built on Flask-SocketIO and asynchronous agents, ensures that administrators are notified of threats instantaneously without the lag associated with traditional polling mechanisms.
- Scalable Agent Architecture: The lightweight `live_agent.py` can be deployed across numerous endpoints without requiring heavy ML libraries locally, centralizing the heavy lifting on the server.

Limitations:
- High Resource Consumption: Training the ML models on the SMOTE-augmented CICIDS2017 dataset requires significant RAM, necessitating subsampling strategies (e.g., capping RF at 500k rows) to prevent Out-Of-Memory (OOM) errors on standard hardware. Running Snort 3 and the Flask server concurrently also demands a multi-core CPU environment for optimal performance.
- Feature Extraction Overhead: While Snort analyzes packets directly, the ML model requires the `live_agent` to construct 40+ statistical features over a time window (e.g., 2 seconds). This inherently introduces a slight delay in ML detection compared to instant packet signature matching.
- Maintenance Overhead: The signature-based component still relies on continuous manual updates to its rule sets (`local.rules`) to remain effective against the latest specific exploits, requiring ongoing administrative effort.
